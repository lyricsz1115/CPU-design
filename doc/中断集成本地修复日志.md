1. `pipeline_cpu_top.v` 重复声明 `ex_div_result`，Vivado 无法编译 CPU。
2. `editable_pipeline_system_top.v` 使用了未显式声明的 `trap_taken`。
3. 外部中断锁存后没有确认清除路径。
4. 中断可在 DIV/Cache 事务未完成时进入，精确中断语义不完整。
5. MRET 在 EX 和后续 `shadow_restore` 阶段可能重复重定向 PC。
6. 一键检查脚本的源文件列表和最终综合顶层不正确。

### 步骤 4：收敛中断接受边界、影子捕获与 MRET

修改文件：

- `src/pipeline_cycle/pipeline_cpu_top.v`
- `src/extension_components/trap_csr_unit.v`

修改内容：

1. 将异步中断请求与真正接受分开，新增 `interrupt_accept`。
2. Cache、DIV 或 debug 暂停期间不接受 trap。
3. EX 正在执行 Load、Store、Branch、JAL、CSR、MRET 或 ECALL 时，异步 IRQ 等待到稳定指令边界。
4. 只有 IF/ID 有效时接受异步 IRQ，保证 `mepc` 对应明确的恢复 PC。
5. 影子捕获增加 EX、MEM、WB 到寄存器当前值的优先选择。
6. MRET 改为 EX 阶段单次提交：恢复影子、恢复 MIE、重定向 PC 使用同一个事件。
7. 删除 `shadow_restore` 对 PC 的第二次重定向和旧 MRET 移位跟踪。
8. MRET 提交时同时清除 IF/ID 与 ID/EX 中的年轻指令。

影子值优先级：

```text
EX 最新结果 > MEM 最新结果 > WB 写回结果 > regfile 当前值
```

设计边界：

- 这是保持现有五级流水线结构的最小侵入方案，不新增完整流水线排空状态机。
- 异步 IRQ 会延迟若干周期，但锁存请求保证不会丢失。
- ECALL 仍以同步异常方式在 EX 接受，但必须等待 Cache/DIV 暂停解除。

检查结果：

- 全部 RTL 通过 `xvlog` 静态分析。
- `uart_editable_pipeline_system_top`、`editable_pipeline_system_top` 和 `tb_trap` 均通过 elaboration。
- 最终 UART 顶层第二次综合通过：0 error、0 critical warning。
- `report_drc` 未发现多驱动；此前的 `trap_taken_out -> debug_stall` 组合环已经消失。
- 本步骤仍未运行功能仿真和上板测试。

复查修正：

- 初版曾将 ECALL 接受条件写成 `!id_ex_flush`。
- 因为 ECALL 自身会产生 `trap_taken` 并进一步产生 `id_ex_flush`，该条件会形成组合自反馈。
- 已删除该条件；ECALL 只由 `id_ex_valid && trap_stage_ready` 限定。
- regfile 已确认以影子恢复为高优先级，同周期普通 WB 不会覆盖 x1/x2/x5/x6/x7 的恢复值。

新增检查工具：

- `vivado/check_final_uart_synthesis.tcl`
- 只综合最终 UART 顶层，不运行仿真、实现或 bitstream。
- 用于发现组合环、多驱动、缺失模块和约束端口错误。

综合复查发现并修复：

- 首次综合完成但报告 2 条组合时序环，路径经过 `trap_taken_out` 和 `debug_stall`。
- 原因是顶层用组合 `trap_taken` 产生 stall，同时 CPU 用 stall 限制 trap 接受。
- 已将 debug stall 改为只依赖时钟锁存的 `debug_frozen`。
- trap 当拍先把 PC 重定向到 `mtvec`，下一拍再冻结 ISR 入口。
- 第二次综合已确认 timing loop 消失。

综合报告边界：

- DRC 剩余警告为配置电压属性未设置，以及乘法 DSP 未加入输入/输出流水寄存器；不是中断信号多驱动或组合环。
- 资源估算：15501 LUT、14081 Register、10 DSP、0 Block RAM Tile。
- 100 MHz 约束下静态时序未通过：WNS = -14.388 ns，TNS = -19188.152 ns。
- 最差路径包含 4 个 DSP48E1 和流水线控制使能，主要来自原有组合乘法与控制路径；当前代码直接使用 100 MHz 系统时钟驱动 CPU，因此不能把该结果解释为分频时钟约束误报。
- 本轮只收敛中断集成，不擅自改变乘法流水级或系统主频。后续验收必须在“降低 CPU 时钟”与“给乘法/控制路径增加流水级”中选择一种时序收敛方案，并重新运行 implementation timing。

接口清理：

- 删除 `trap_csr_unit` 中旧 MRET 移位跟踪遗留且已无用途的 `ex_mem_valid`、`ex_mem_flush`、`mem_wb_valid` 输入。
- 更新模块头部与影子寄存器注释，使其与 EX 阶段单次 MRET、EX/MEM/WB 快照前递逻辑一致。
- 全仓调用搜索确认 `trap_csr_unit` 只有 CPU 内一处实例，删除端口后没有遗漏连接；按约定未为本次纯接口清理重复运行耗时综合或仿真。
