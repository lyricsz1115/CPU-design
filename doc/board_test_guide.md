## 1. 板子控件

| 控件 | 位置 | 功能 |
|------|------|------|
| SW7~SW0 | 8 个拨码开关 | 输入当前字节（↑=1, ↓=0） |
| S1 | btn_write | **写入**当前字节（Byte0→1→2→3，写完跳到下一条） |
| S2 | btn_next | 跳到下一条指令（跳过未写完的字节） |
| S3 | btn_clear | **清空**，回到录入模式，地址归零 |
| S4 | btn_run | **执行**程序 |
| rst_btn | 按键 | 复位 |
| LED7~LED0 | 8 灯 | 显示结果（录入模式闪烁，运行模式显示结果） |

## 2. LED 解读

运行模式下直接拨动 SW7、SW6 选择显示，不需要再按按钮：

| SW7:6 | LED 内容 |
|-------|----------|
| `00` | 程序结果；结果为 0 时回退显示 `{1'b1, PC[6:0]}` |
| `01` | Cache 访问次数低 8 位 |
| `10` | Cache 命中次数低 8 位 |
| `11` | Cache 缺失次数低 8 位 |

> 查看普通程序结果时必须将 SW7:6 调成 `00`。如果程序确实写入了最高位为 1 的结果（例如负数低 8 位），LED 最高位也会为 1，因此不能只凭最高位判断是否正在显示 PC。

### 4.3 第 2~6 轮：修改写回 DMEM[0] 的源寄存器

按 S3 (CLEAR) 后重新录入全部，只修改 `[3] sw` 指令的 B2。B1 必须保持 `20`，以保证写地址仍为 0：

| 3 | `23 20 50 00` | B2=`50` | `00000010` | REM=2 |
| 4 | `23 20 60 00` | B2=`60` | `00010100` | ANDI=20 |
| 5 | `23 20 70 00` | B2=`70` | `00110000` | SLLI=48 |

## 6. Cache 强验证（9 条指令）

仅执行 Store 会统计 Cache 访问和缺失，但无法证明读缺失填充与命中路径正确。本用例通过重复 Load 和同一 Cache 行内 Store，同时验证流水线暂停、行填充、读命中和写命中。

| 地址 | 机器码 | 录入字节 (B0 B1 B2 B3) | 汇编 |
|------|--------|-------------------------|------|
| 0 | `01400093` | `93 00 40 01` | `addi x1,x0,20` |
| 1 | `00300113` | `13 01 30 00` | `addi x2,x0,3` |
| 2 | `022081B3` | `B3 81 20 02` | `mul x3,x1,x2` |
| 3 | `00302023` | `23 20 30 00` | `sw x3,0(x0)` |
| 4 | `00002203` | `03 22 00 00` | `lw x4,0(x0)` |
| 5 | `00002283` | `83 22 00 00` | `lw x5,0(x0)` |
| 6 | `00128313` | `13 83 12 00` | `addi x6,x5,1` |
| 7 | `00602223` | `23 22 60 00` | `sw x6,4(x0)` |
| 8 | `0000006F` | `6F 00 00 00` | `jal x0,0` |

输入完成后按 S4，直接拨动 SW7:6，预期如下：

| SW7:6 | 预期 LED | 判定 |
|-------|----------|------|
| `00` | `00111100` (60) | MUL、Store 和结果显示正确 |
| `01` | `00000100` (4) | 共发生 4 次 Cache 访问 |
| `10` | `00000010` (2) | 第二次 Load 和地址 4 Store 命中 |
| `11` | `00000010` (2) | 首次 Store、首次 Load 缺失 |

访问顺序：

```text
sw  地址0：写缺失，写直达且不分配
lw  地址0：读缺失，填充包含地址0/4/8/12的Cache行
lw  地址0：读命中
sw  地址4：写命中，同时写直达后端
```

只有得到 `60 / 4 / 2 / 2`，才能证明 Cache 不只是被综合进设计，而且读缺失、行填充、命中和流水线恢复路径都被实际执行。

## 8. 故障排查

| 现象 | 原因 | 解决 |
|------|------|------|
| SW7:6=`00` 且结果为 `1xxxxxxx` | 可能是负数结果，也可能是 dmem[0]=0 时的 PC 回退显示 | 先核对预期结果，再核对 Byte0~3 是否与 §7 表一致 |
| SW7:6 切换后 LED 不变化 | 烧录了旧顶层或 Vivado 引用了另一份源码 | 检查 Source File Properties、Reset Run 后重新生成 bitstream |
| 除法和预期不对 | 没加 div_unit.v | 确保工程包含 `src/extension_components/div_unit.v` |
| WNS < 0 | alu.v 仍有组合除法或并行乘法器过多 | 确认 `alu.v` DIV case=`y=32'b0` 且仅一个共享乘法器 |
| WNS < 0（>=10ns 违规）| alu.v BUG #6：三个并行乘法器 | 确认 `alu.v` 用 `mul_a_s * mul_b_s` 替代三个 `wire mul_*` |
| MUL+SW 正确但含 DIV 程序失败 | BUG #5: ex_mem_flush 竞争 | 确认 `pipeline_cpu_top.v` 中 `ex_mem_en = ~cache_stall & ~div_stall` |
| 测试 B 失败（SW-before-DIV=0 但 C 通过） | 未更新比特流，BUG #5 未修复 | Vivado 重新 Generate Bitstream |
| 按 S1 没反应 | 按键太快 <10ms | 按住再松手，等待 LED 变化 |
