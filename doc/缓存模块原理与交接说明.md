## 2. 默认 Cache 规格

| 项目 | 默认值 |
| --- | --- |
| 地址宽度 | 32 位 |
| 数据宽度 | 32 位 |
| Cache 容量 | 256 B |
| 组数 | 8 组 |
| 路数 | 2 路 |
| 每行大小 | 16 B |
| 每行 word 数 | 4 |
| 替换策略 | 每组 1 bit 真 LRU（两路情况下足够） |
| 读 miss | 整行回填，即依次读取 4 个 word |
| 写策略 | Write-through，写直达 |
| 写 miss | No-write-allocate，写未命中不分配 |
| 并发能力 | Blocking，一次只处理一个 CPU 请求 |
| 统计 | access、hit、miss 三个 32 位计数器 |

容量计算：

```text
8 sets × 2 ways × 4 words × 4 bytes = 256 bytes
```

## 3. 为什么选择当前策略

### 3.1 两路组相联

直接映射 Cache 中，一个地址只能进入唯一 Cache line，具有相同 index 的数据会频繁互相替换。两路组相联允许每个 set 保存两个不同 tag，在复杂度仍较低的情况下减少冲突 miss。

### 3.2 Write-through

写命中时同时修改 Cache 和后端内存。这样 Cache 中没有 dirty line，不需要处理脏行写回，状态机和后续流水线接入都更简单。

代价是每次写都要等待后端内存确认，写密集程序的性能不如 write-back。后续可增加 write buffer 缓解。

### 3.3 No-write-allocate

写 miss 不读取整行，只把这个 word 写入后端内存。下一次读取同一地址仍然 miss，再通过正常读 miss 回填。

该策略与 write-through 配合简单，适合当前课程设计阶段。

### 3.4 Blocking Cache

Cache 在完成当前访问前不接受第二个访问。CPU 侧通过 `req_ready` 和 `busy` 得知状态。它不支持多个 outstanding miss，但控制明确，适合五级顺序流水 CPU。

## 4. 地址如何拆分

默认配置中，一行有 4 个 32 位 word，所以一行是 16 B；共有 8 个 set。

32 位地址拆分如下：

```text
31                         7 6       4 3       2 1       0
+---------------------------+---------+---------+---------+
|            tag            |  index  |  word   |  byte   |
+---------------------------+---------+---------+---------+
           25 bit              3 bit     2 bit     2 bit
```

- `byte offset = addr[1:0]`：选择 word 内的 byte。当前只支持完整 word，所以最低两位实际应为 0。
- `word offset = addr[3:2]`：选择一行中的 4 个 word 之一。
- `index = addr[6:4]`：选择 8 个 set 之一。
- `tag = addr[31:7]`：判断该 way 中保存的是哪个内存块。

例如：

```text
0x00000000、0x00000004、0x00000008、0x0000000c
```

属于同一个 Cache line，只是 word offset 不同。

以下地址的 index 都为 0，但 tag 不同，因此会进入同一个 set 的两路并发生替换竞争：

```text
0x00000000
0x00000080
0x00000100
0x00000180
```

## 5. 内部存储结构

每个 way 各自包含：

```text
tag_way0[set] / tag_way1[set]
valid_way0[set] / valid_way1[set]
data_way0[set * WORDS_PER_LINE + word]
data_way1[set * WORDS_PER_LINE + word]
```

每个 set 还有一个 `lru_way`：

```text
lru_way=0 -> 下次需要替换时优先替换 way 0
lru_way=1 -> 下次需要替换时优先替换 way 1
```

访问 way 0 后，way 1 变为最近最少使用，因此写 `lru_way=1`；访问 way 1 后写 `lru_way=0`。

选择 miss victim 时的优先级：

1. way 0 无效，选择 way 0。
2. 否则 way 1 无效，选择 way 1。
3. 两路都有效，按照 `lru_way` 替换。

## 6. CPU 侧接口

```verilog
input  req_valid;
output req_ready;
input  req_write;
input  [31:0] req_addr;
input  [31:0] req_wdata;

output resp_valid;
output [31:0] resp_rdata;
output resp_hit;
output busy;
```

### 6.1 请求握手

只有 `req_valid && req_ready` 同时为 1 的上升沿，Cache 才接收新请求。接收后会锁存地址、读写方向和写数据，并让 `req_ready=0`、`busy=1`。

请求方在请求被接收后必须撤销 `req_valid`，不能一直保持到响应结束，否则 Cache 回到 IDLE 后会把它当成第二个请求。

### 6.2 响应

访问完成时，`resp_valid` 拉高一个周期：

- 读请求：`resp_rdata` 是读取结果。
- 写请求：`resp_rdata` 固定为 0，表示完成确认。
- `resp_hit=1` 表示原始访问命中。
- `resp_hit=0` 表示原始访问未命中，即使 miss 后已经完成回填或写后端。

## 7. 后端内存接口

```verilog
output mem_req_valid;
input  mem_req_ready;
output mem_req_write;
output [31:0] mem_req_addr;
output [31:0] mem_req_wdata;

input  mem_resp_valid;
input  [31:0] mem_resp_rdata;
```

请求在 `mem_req_valid && mem_req_ready` 的上升沿被后端接受。Cache 随后撤销请求并等待 `mem_resp_valid`。

约定后端不能在接受请求的同一个上升沿立即返回 `mem_resp_valid`，应至少延迟一个周期。testbench 的内存模型默认延迟两个周期。

写事务也必须返回 `mem_resp_valid`，它作为写完成确认；写响应的数据内容不使用。

## 8. 状态机

状态流如下：

```text
IDLE
  |
  | 接收 CPU 请求
  v
LOOKUP
  |-- 读 hit ------------------------------> IDLE + resp
  |-- 写 hit --> WRITE_REQ --> WRITE_WAIT -> IDLE + resp
  |-- 写 miss -> WRITE_REQ --> WRITE_WAIT -> IDLE + resp
  `-- 读 miss -> REFILL_REQ <-> REFILL_WAIT -> IDLE + resp
```

### 8.1 `STATE_IDLE`

- `req_ready=1`。
- 接收并锁存一个 CPU 请求。
- `access_count` 加 1。
- 转入 LOOKUP。

### 8.2 `STATE_LOOKUP`

同时比较两个 way 的 valid 和 tag。

读命中：

- `hit_count` 加 1。
- 组合读取对应 word。
- 更新 LRU。
- 产生响应并回到 IDLE。

写命中：

- `hit_count` 加 1。
- 立即更新命中 way 中的数据。
- 更新 LRU。
- 进入后端写事务，收到写确认后响应 CPU。

读未命中：

- `miss_count` 加 1。
- 选择无效 way 或 LRU victim。
- 从 word 0 开始回填整行。

写未命中：

- `miss_count` 加 1。
- 不修改 tag、valid、data 和 LRU。
- 直接写后端内存。

### 8.3 `STATE_WRITE_REQ/WAIT`

REQ 状态保持内存写请求，直到后端 `mem_req_ready=1`。WAIT 状态等待写完成响应。

由于是 write-through，写命中也必须等待后端确认后才能向 CPU 返回完成。

### 8.4 `STATE_REFILL_REQ/WAIT`

回填从 line base 开始，依次发出：

```text
line_base + 0
line_base + 4
line_base + 8
line_base + 12
```

每收到一个 `mem_resp_valid` 就写入 victim way 的对应 word。第四个 word 返回后：

- 写入新 tag。
- 设置 valid。
- 更新 LRU，使另一 way 成为下一个 victim。
- 返回原始读请求所需要的 word。

在整行填完前不会对外声明响应完成，因此不会让 CPU 读到半行有效数据。

## 9. 性能计数器

```verilog
access_count
hit_count
miss_count
```

- 每接收一个 CPU 请求，`access_count` 加 1。
- LOOKUP 命中时 `hit_count` 加 1。
- LOOKUP 未命中时 `miss_count` 加 1。
- 复位时三个计数器清零。
- 32 位自然溢出，没有饱和逻辑。

始终应满足：

```text
access_count = hit_count + miss_count
hit_rate = hit_count / access_count
miss_rate = miss_count / access_count
AMAT = hit_time + miss_rate * miss_penalty
```

## 10. Testbench 结构

`tb_data_cache.v` 内含两个 module：

### 10.1 `tb_data_cache`

它产生 CPU 请求并检查返回数据、命中状态、LRU、写策略和统计计数。

主要 task：

| task | 作用 |
| --- | --- |
| `reset_cache` | 复位 Cache 和握手状态 |
| `cache_read` | 发出一次读请求，等待响应 |
| `cache_write` | 发出一次写请求，等待写确认 |
| `require_read` | 检查读数据和 hit/miss 是否符合预期 |
| `require_counts` | 检查三个统计计数器 |

### 10.2 `cache_memory_model`

这是仅用于仿真的后端内存模型：

- 1024 个 32 位 word。
- 初始 `mem[i] = 0x10000000 + i`，便于预测读取结果。
- 一次只接收一个事务。
- 默认响应延迟为两个周期。
- 写事务修改 `mem` 并返回完成脉冲。

它不是最终的板上存储器模块。接入 CPU 时应替换成 BRAM 控制器、现有数据存储器适配器或其他总线从设备。

## 11. 已覆盖的访问模式

### 11.1 顺序访问 64 个 word

每行 4 个 word，所以每行第一个 word miss，后面三个 hit：

```text
access=64, hit=48, miss=16, hit rate=75.00%
```

### 11.2 重复访问 4 个 word

4 个 word 全部位于同一行。第一次访问 miss 并整行回填，后面 31 次全部 hit：

```text
access=32, hit=31, miss=1, hit rate=96.875%
```

### 11.3 以 16 B 为步长访问

每次只取一个新 line 的一个 word，第一次遍历没有空间重用：

```text
access=16, hit=0, miss=16, hit rate=0%
```

### 11.4 四次循环访问 16 个 word

工作集只有 4 行，可以放入 Cache：

```text
access=64, hit=60, miss=4, hit rate=93.75%
```

### 11.5 确定性伪随机访问

使用固定初值的 8 位 LFSR 生成 64 次访问。testbench 检查总访问数以及：

```text
access_count == hit_count + miss_count
```

运行日志中的 hit/miss 可以作为以后修改容量、行大小和路数时的对比基准。

## 12. LRU 验证过程

testbench 使用映射到 set 0 的 A、B、C：

```text
A = 0x00000000
B = 0x00000080
C = 0x00000100
```

验证顺序：

1. 读 A，miss，进入 way 0。
2. 读 A+4，命中同一行。
3. 读 B，miss，进入 way 1。
4. 再读 A，使 A 成为 MRU、B 成为 LRU。
5. 读 C，应该替换 B。
6. 再读 A，必须 hit，证明 A 没被错误替换。
7. 再读 B，必须 miss，证明 LRU 确实选择了 B。

## 13. 写策略验证

### 13.1 写命中

先确保 A 所在行已经在 Cache，然后执行：

```text
write 0x00000004 = 0xdeadbeef
```

检查：

- `resp_hit=1`。
- Cache 内对应 word 更新。
- 后端 `mem[1]` 同时变为 `0xdeadbeef`。
- 随后读取同一地址命中并返回 `0xdeadbeef`。

### 13.2 写未命中

对不在 Cache 中的地址写：

```text

```

检查：

- `resp_hit=0`。
- 后端内存正确更新。
- Cache 不分配该行。
- 紧接着读取该地址仍然 miss，然后回填并返回 `0xcafebabe`。

## 15. 流水线接入实现

当前数据通路为：

```text
EX/MEM 有效地址和写数据
          |
          v
       data_cache
          |
          v
后端数据存储器或内存总线
```

实现没有直接删除 `dmem`，而是在 Cache 和现有存储接口之间加入 `cache_memory_adapter`。

已经完成：

1. 用 `ex_mem_valid && (mem_read || mem_write)` 产生一次 Cache 请求。
2. 请求被 `req_ready` 接收后记录“该 MEM 指令已经发出”，防止 stall 期间重复请求。
3. `resp_valid` 到来前冻结 PC、IF/ID、ID/EX、EX/MEM，并阻止 MEM/WB 接收未完成数据。
4. Cache 读响应进入 `mem_wb_reg.mem_data_in`。
5. 写请求只有收到响应后才允许该 store 离开 MEM。
6. 所有寄存器写、内存写和提交计数必须由对应 stage valid 门控。
7. 将 `cache_stall` 与 `div_stall` 统一到 `front_stall`。

当前主要控制关系为：

```text
reset
  > 分支/异常恢复
  > Cache miss stall 或 muldiv busy
  > load-use stall
  > 正常推进
```

Cache 等待时 EX/MEM 保持当前访存指令；除法等待时 EX/MEM 注入 bubble，让更老的指令继续退休。若 Cache 中有更老的 miss 而 EX 同时计算除法，除法结果会锁存到 `div_result_latched`，直到 Cache 允许该除法进入 EX/MEM。

## 16. MMIO 必须绕过 Cache

当前系统把 LED、SW 和性能计数器映射在 `0x10000000` 附近。此类地址有读写副作用或代表实时硬件状态，不能被普通 Cache 保存。

当前按地址选择：

```text
0x00000000 ~ 普通内存范围 -> data_cache
0x10000000 ~ MMIO 范围    -> io_bus，绕过 Cache
```

特别是：

- SW 必须每次读实时值，不能命中旧 Cache line。
- LED 写不能只更新 Cache，必须真正到达 I/O 寄存器。
- 未来 UART_RXDATA 的读取可能具有“读后弹出 FIFO”副作用，绝不能缓存。

## 17. 与现有 `dmem` 的适配

当前 `dmem.v` 是组合读、同步写，没有 request/response 握手。Cache 后端则要求请求握手和完成响应。

当前使用的独立适配器为：

```text
cache_memory_adapter
├── 接收 mem_req_valid/mem_req_ready
├── 对 dmem 发出一次读或写
└── 下一周期产生 mem_resp_valid
```

如果后续把 dmem 改成 BRAM，同步读本来就有一拍延迟，适配器仍然适用。

## 18. 当前限制

1. 只支持完整 32 位 word，没有 byte enable，因此不能直接支持 SB/SH/LB/LH。
2. 固定两路；参数只改变 set 和 line word 数，不改变 ways。
3. `NUM_SETS` 和 `WORDS_PER_LINE` 必须是 2 的幂，并且当前实现要求二者至少为 2。
4. 数据宽度虽然是参数，但地址每次固定加 4，实际按 32 位 word 设计。
5. 没有 dirty bit、write-back 和 write buffer。
6. 没有预取和 non-blocking miss。
7. 没有单独的软件 invalidate/flush 输入，只能通过复位清 valid。
8. 复位循环清除全部 data/tag 数组，可能影响 Vivado 对 BRAM 的推断；当前容量小，通常会使用 LUT/FF。
9. 后端没有错误响应，假定所有内存事务最终成功。
10. 后端必须至少在接受请求后的下一周期返回，不能零周期响应。
11. 未检查非对齐地址，默认请求地址最低两位为 0。
12. 已接入 CPU，但 Cache 数据阵列和现有 io_bus/dmem 仍主要映射为 LUT/FF，没有完成 BRAM 优化。

```text
tb_data_cache       PASS：独立 Cache、LRU、refill、写策略、统计
tb_mul_div          PASS：流水 MUL/DIV/REM、I-type、BEQ 和单周期回归
tb_cached_pipeline  PASS：Cache demo=6/4/2，Cache+MUL/DIV 结果正确
tb_pipeline         PASS：forwarding、load-use、branch prediction/flush
tb_io_system        PASS：内存映射 LED 和性能计数
tb_editable_loader  PASS：SW/按钮机器码装载
```

远端原始 `tb_mul_div` 拉取后曾出现流水线 6 项失败和持续 stall。修复内容：

- 除法指令从启动周期开始保持在 EX，不能提前进入 EX/MEM。
- `div_unit` 的 DONE 状态输出一个完整周期的 `done`。
- 除零等特殊结果使用独立完成状态，避免被普通 DONE 计算覆盖。
- Cache miss 阻挡除法写回时锁存除法结果。
- stage valid 门控寄存器写回和 forwarding。

随后新增了完整 LUI 的 `alu_a_zero` 流水传递，并在板测程序加入 `0x10000000` MMIO LED 写，用于验证 MMIO 绕过。该最后增量已通过 `xvlog` 编译和静态代码核对；由于用户会话中遗留的 Vivado 进程导致 `xelab` 启动器空日志退出，仍应在下载前重新运行一次 `tb_cached_pipeline`。

## 22. 板级测试方法

Vivado 顶层：

```text
cached_pipeline_minisys_top
```

约束：

```text
xdc/cached_pipeline_minisys_template.xdc
```

内置程序为 `PROGRAM_ID=8`，无需 `$readmemh`。按 S6 复位后，通过 SW7、SW6 选择 LED：

| SW7:SW6 | LED 内容 | 预期值 |
| --- | --- | --- |
| `00` | 程序结果/MMIO LED | `0x55`，即 `01010101` |
| `01` | access_count 低 8 位 | `6`，即 `00000110` |
| `10` | hit_count 低 8 位 | `4`，即 `00000100` |
| `11` | miss_count 低 8 位 | `2`，即 `00000010` |

综合日志显示 imem 使用 distributed RAM，Cache、dmem 和 io_bus 的大部分存储仍使用 LUT/FF。当前容量可以放入 xc7a100t，但资源效率不是最终状态。由于源码在该 DCP 后还有小幅更新，资源数字仅供参考，必须从零重新综合当前版本。

```powershell
& 'E:\Vivado\Vivado\2018.3\bin\vivado.bat' -mode batch -source 'vivado\create_cached_pipeline_project.tcl'
```

必须确认 implementation 完成、WNS 非负且生成 `cached_pipeline_minisys_top.bit` 后，再通过 Hardware Manager 下载。
