## 4. RTL 修改记录

修改文件：

```text
src/editable_pipeline_system_top.v
```

### 4.1 增加 Cache 统计信号

新增三个 32 位连线：

```verilog
wire [31:0] debug_cache_access_count;
wire [31:0] debug_cache_hit_count;
wire [31:0] debug_cache_miss_count;
```

分别表示 CPU 发起的可缓存数据访问次数、命中次数和缺失次数。

### 4.2 在可编辑顶层启用数据 Cache

`pipeline_cpu_top` 的参数增加：

```verilog
.ENABLE_DATA_CACHE(1),
.CACHE_NUM_SETS(8),
.CACHE_WORDS_PER_LINE(4)
```

修改后的数据访问路径为：

```text
流水线 CPU
  -> data_cache
  -> cache_memory_adapter
  -> external data bus
  -> io_bus
```

Cache 为二路组相联结构，共 8 组，每个 Cache 行包含 4 个 32 位字。

### 4.3 接出 Cache 统计端口

CPU 实例增加以下端口连接：

```verilog
.debug_cache_access_count(debug_cache_access_count),
.debug_cache_hit_count(debug_cache_hit_count),
.debug_cache_miss_count(debug_cache_miss_count),
```

这样统计信号不会停留在 CPU 内部，而是可以由上板顶层送到 LED。

### 4.4 增加运行状态 LED 显示选择

原程序结果显示逻辑保留为：

```verilog
wire [7:0] result_display =
    (bus_led != 8'b0) ? bus_led : {1'b1, debug_pc[6:0]};
```

运行状态下新增组合选择：

```verilog
always @(*) begin
    case (sw[7:6])
        2'b00: run_display = result_display;
        2'b01: run_display = debug_cache_access_count[7:0];
        2'b10: run_display = debug_cache_hit_count[7:0];
        2'b11: run_display = debug_cache_miss_count[7:0];
        default: run_display = result_display;
    endcase
end
```

显示含义如下：

| SW7 | SW6 | LED 内容 |
|---:|---:|---|
| 0 | 0 | 程序结果 |
| 0 | 1 | Cache 访问次数低 8 位 |
| 1 | 0 | Cache 命中次数低 8 位 |
| 1 | 1 | Cache 缺失次数低 8 位 |

进入运行模式后，拨动 SW7 和 SW6 即可立即切换显示，不需要按任何按钮。

## 5. 手动输入和按钮功能

本次没有修改 `instr_loader`，原有功能保持不变：

| 板上按钮 | RTL 端口 | 功能 |
|---|---|---|
| S1 | `btn_write` | 写入当前 `SW[7:0]` 字节 |
| S2 | `btn_next` | 跳到下一条指令地址 |
| S3 | `btn_clear` | 返回加载模式并清除加载位置 |
| S4 | `btn_run` | 进入运行模式，从 PC=0 开始执行 |

每条机器指令按照低字节优先顺序输入。第 4 个字节写入后，加载器会自动写入完整 32 位指令并将指令序号加一。连续输入程序时不应在每条指令后额外按 S2。

加载阶段仍使用完整 `SW[7:0]` 输入机器码。只有进入运行模式后，`SW[7:6]` 才用于选择 LED 显示内容，所以 Cache 统计显示不会破坏手动编程功能。

## 11. Cache 与流水线冲突及当前解决机制

### 11.1 冲突的根源

普通五级流水线默认每一级每个周期都能向后推进，但 Cache 访问不是固定单周期操作：

- 命中需要经过请求接收和标签查询。
- 读缺失需要从后端存储器逐字填充一整条 Cache 行。
- 写操作采用写直达，需要等待后端写请求完成。

如果 MEM 级尚未得到结果，流水线却继续推进，会发生以下错误：

- Load 在数据有效前进入 WB，向寄存器写入旧值或 0。
- EX/MEM 被下一条指令覆盖，丢失正在等待的地址和写数据。
- PC 和前级继续前进，使后续指令使用尚未完成的 Load 结果。
- 同一条停留在 MEM 级的指令可能每周期重复向 Cache 发起请求。

因此 Cache 和流水线的核心矛盾是“流水线固定节拍推进”与“存储访问可变延迟”之间的冲突。

### 11.2 请求和响应握手

`pipeline_cpu_top` 与 `data_cache` 之间使用以下握手信号：

```text
cache_req_valid：CPU 当前有新的 Cache 请求
cache_req_ready：Cache 可以接收请求
cache_resp_valid：Cache 已完成当前请求
cache_resp_rdata：读请求返回的数据
```

只有有效的 MEM 级 Load/Store 且地址属于低 1 KiB 数据存储区时，访问才进入 Cache：

```verilog
assign mem_access = ex_mem_valid && (mem_mem_read || mem_mem_write);
assign mem_cacheable = (mem_alu_result[31:10] == 22'b0);
assign cached_mem_access = ENABLE_DATA_CACHE && mem_access && mem_cacheable;
```

Cache 等待条件为：

```verilog
assign cache_stall = cached_mem_access && !cache_resp_valid;
```

含义是：只要当前 MEM 级访问属于 Cache，但 Cache 还没有返回完成信号，流水线就必须暂停。

### 11.3 暂停期间如何保持流水线状态

Cache 暂停被并入统一前端暂停：

```verilog
assign front_stall = div_stall | cache_stall;
```

`front_stall` 送入 `hazard_unit`，暂停期间执行以下动作：

```text
PC：保持，不取下一地址
IF/ID：保持，不接收下一条指令
ID/EX：保持，不让后续指令进入 EX
EX/MEM：保持当前访存指令及地址、写数据和控制信号
MEM/WB：清空，防止等待中的指令提前提交
```

对应 RTL 控制为：

```verilog
assign id_ex_en  = ~front_stall;
assign ex_mem_en = ~cache_stall;
```

EX/MEM 有效位在 Cache 等待期间保持：

```verilog
end else if (cache_stall) begin
    ex_mem_valid <= ex_mem_valid;
end
```

MEM/WB 在等待期间执行 flush：

```verilog
mem_wb_reg u_mem_wb(
    .en(1'b1),
    .flush(cache_stall),
    ...
);
```

同时 `mem_wb_valid` 被置为 0，因此等待周期不会发生错误的寄存器写回，也不会把同一条访存指令重复计算为已退休指令。

### 11.4 防止重复发送同一请求

由于 Cache miss 时 EX/MEM 会保持多个周期，如果直接使用 `cached_mem_access` 作为请求有效信号，同一条指令会在每个等待周期重复发送。

当前使用 `cache_req_sent` 记录本次请求是否已经被 Cache 接收：

```verilog
assign cache_req_valid = cached_mem_access && !cache_req_sent;
```

当 `cache_req_valid && cache_req_ready` 成立时置位；当访问结束或 `cache_resp_valid` 返回时清零。这样一条 Load/Store 在整个等待过程中只会被 Cache 接收一次。

### 11.5 Cache 内部状态机

`data_cache` 使用状态机把可变延迟访问转换成明确的请求/完成协议：

```text
STATE_IDLE
  -> STATE_LOOKUP
  -> 命中读：返回 resp_valid
  -> 写操作：STATE_WRITE_REQ -> STATE_WRITE_WAIT
  -> 读缺失：STATE_REFILL_REQ -> STATE_REFILL_WAIT
```

读命中时直接从对应路和字偏移返回数据。读缺失时从后端依次读取 `WORDS_PER_LINE=4` 个字，全部填充完成后才向 CPU 拉高 `resp_valid`。

写策略为写直达：

- 写命中同时更新 Cache 中的数据副本，并把数据写入后端。
- 写缺失不分配 Cache 行，直接写入后端。
- CPU 必须等到后端响应后才解除 `cache_stall`。

写直达避免了 dirty bit 和脏行替换问题，也保证 `io_bus.mem[0]` 最终能得到 Store 的结果，代价是每次写操作都需要等待后端写入完成。

### 11.6 后端适配器的作用

`cache_memory_adapter` 位于 Cache 和单周期后端总线之间。它用 `pending` 保存一次后端事务：

```text
req_ready = !pending
接收请求后 pending = 1
下一周期 resp_valid = pending
```

该适配器把 Cache 的请求/响应接口转换为 `io_bus` 使用的 `mem_read`、`mem_write`、地址和写数据接口，使 Cache 不依赖后端必须在同一周期完成握手。

### 11.7 请求完成后的恢复过程

当 `cache_resp_valid=1` 时：

1. `cache_stall` 变为 0。
2. Load 返回值通过 `cache_resp_rdata` 送入 MEM/WB。
3. 当前访存指令允许进入 WB。
4. PC、IF/ID、ID/EX 和 EX/MEM 恢复推进。
5. `cache_req_sent` 清零，为下一次访问做准备。

Load 后紧跟使用该寄存器的指令时，原有 load-use hazard 和转发逻辑继续生效。Cache miss 只会延长 MEM 级停留时间，不会绕过原来的数据相关处理。

### 11.8 与多周期除法的协调

Cache 暂停和除法暂停统一合成为：

```verilog
front_stall = div_stall | cache_stall;
```

因此无论 EX 级正在等待除法器，还是 MEM 级正在等待 Cache，PC 和前端流水级都会停止推进。Cache 暂停还会单独保持 EX/MEM，以保证正在进行的访存事务不被覆盖。

### 11.9 可缓存区与外设旁路

当前仅缓存满足以下条件的地址：

```verilog
mem_alu_result[31:10] == 22'b0
```

即地址 `0x00000000` 到 `0x000003FF`。LED、拨码开关和性能计数器等 `0x10000000` 区域的内存映射 I/O 不进入 Cache，而是通过 `direct_mem_read` 或 `direct_mem_write` 直接访问后端。

这避免了读取拨码开关时拿到旧缓存值，也避免 LED 写入停留在 Cache 中而没有送到真实 I/O。

### 11.10 当前方案的边界

当前机制解决的是单核、顺序提交流水线与一级数据 Cache 之间的延迟和数据保持问题，边界如下：

- 没有多核 Cache 一致性协议。
- 没有 DMA 或其他总线主设备修改内存后的 Cache 失效机制。
- 没有异常、中断或总线错误返回处理。
- 写操作采用写直达，不支持 write-back 和 dirty line。
- 后端适配器假设后端事务可以按照当前固定响应方式完成。
- Cache stall 与除法 stall 已统一冻结前端，但后续扩展其他多周期单元时仍需接入同一套有效位和暂停协议。

在当前 Minisys 单核上板范围内，上述机制足以保证 Cache miss、命中和写直达等待期间流水线不会越过未完成的访存指令。
