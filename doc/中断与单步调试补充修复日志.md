# 中断与单步调试补充修复日志

## 修改目标

本次修改针对上板验收中暴露的两个问题：

1. `mini_trap` 测试可以进入中断服务程序，LED 到达 `0xFF`，但不能稳定返回 `0xAA`。
2. `SW8=1` 后普通程序不能从起点按 `S2` 单步观察 PC。

## 问题原因

### 1. mini_trap 停在 0xFF

`mini_trap` 的 ISR 末尾有一条测试指令：

```asm
addi x1, x0, 0xDE
mret
```

这条 `addi` 是故意破坏 x1，用来验证 MRET 时影子寄存器恢复是否真正生效。

旧本地修复版在 MRET 的 EX 阶段立即输出 `shadow_restore`。这样恢复 x1 的时机太早，
ISR 末尾的 `addi x1, x0, 0xDE` 后续到达 WB 时，会再次覆盖 x1。主程序返回后看到的
x1 不再是预期的 `0x5A`，因此验证失败，LED 停留在 ISR 写入的 `0xFF` 或进入失败路径。

### 2. SW8/S2 不能普通单步

原逻辑只在 `trap_taken` 后设置 `debug_frozen`，因此它更像“中断后自动停住”，不是
“普通程序默认冻结，然后按 S2 推进一步”。

## 修改内容

### 1. 分离 MRET 的 PC 跳转和影子寄存器恢复

`pipeline_cpu_top.v` 中，MRET 仍在 EX 阶段产生 `ex_mret_taken`，用于及时重定向 PC：

```verilog
assign ex_mret_taken = ex_is_mret && id_ex_valid &&
                       trap_stage_ready && !id_ex_flush;
```

`trap_csr_unit.v` 中，`shadow_restore` 延后一拍输出：

```verilog
mret_restore_pending <= ex_is_mret && id_ex_valid &&
                        trap_stage_ready && !id_ex_flush;

assign shadow_restore = mret_restore_pending;
```

这样影子寄存器恢复会和 MRET 前一条普通指令的 WB 同拍发生。`regfile.v` 里已有
`shadow_restore` 高于普通 WB 的优先级，因此 ISR 末尾的 `addi x1, 0xDE` 不会覆盖
恢复出来的 `0x5A`。

### 2. 修改 SW8/S2 单步语义

两个顶层都改为：

```verilog
debug_stall = debug_mode && run_mode && !btn_db_posedge;
```

效果：

- `SW8=1` 且 CPU 已运行时，默认冻结流水线。
- 每按一次 `S2`，释放一个时钟周期。
- `SW8=0` 时恢复正常连续运行。
- `SW8=1` 时 `S2` 不作为外部中断，只作为单步按钮。

## 预期验收结果

### mini_trap

运行最小中断测试后：

- `SW7:6=00`，LED 最终应为 `0xAA`。
- 数码管切到 DMEM，`SW7:0=0` 应显示 `0000005A`。
- 数码管切到 DMEM，`SW7:0=1` 应显示 `0000005A`。
- `DMEM[0]` 不应再从 `0x5A` 变成 ISR 末尾测试写入的值。

### 单步

运行任意短程序前或运行后：

- 将 `SW8=1`。
- 数码管显示 PC。
- 每按一次 `S2`，PC 应推进一次。
- 将 `SW8=0` 后，CPU 恢复连续运行。
